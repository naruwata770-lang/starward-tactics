# レビュー観点・基準

レビューは以下の 5 つの観点で評価する。各観点ごとに「何を見るか」と「重要度判定の目安」を示す。

## 1. 正確性 (Correctness)

仕様通りに動作するか、エッジケースを正しく扱っているか。

**チェック対象:**
- ロジックの誤り、計算ミス、境界条件
- null / undefined / 空配列 / 空文字 などの取り扱い
- 非同期処理の競合状態、Promise の取りこぼし
- 型の整合性 (TypeScript の `any` 多用、unsafe cast)
- エラーパスの考慮漏れ

**重要度の目安:**
- **高**: クラッシュ、データ破損、無限ループ、明確な仕様違反
- **中**: 特定条件でしか発生しないバグ、UI の表示崩れ
- **低**: 影響範囲が小さい挙動の差

## 2. セキュリティ (Security)

意図しない情報漏洩や攻撃経路を作っていないか。

**チェック対象:**
- ユーザー入力のバリデーション・サニタイズ
- XSS (`innerHTML`, `dangerouslySetInnerHTML`, `eval`)
- 認証・認可ロジックの抜け
- 機密情報（APIキー、トークン、PII）のログ出力・コミット
- 依存パッケージの既知脆弱性

**重要度の目安:**
- **高**: 認証回避、XSS / SQLi、機密漏洩
- **中**: ログに PII が出る、CSRF 対策の漏れ
- **低**: コメント内の古い TODO など

## 3. 保守性 (Maintainability)

将来の自分や他人が読みやすく、修正しやすいか。

**チェック対象:**
- 命名 (変数・関数・コンポーネント) の妥当性
- 関数・コンポーネントの責務分離 (1 関数 1 責務)
- マジックナンバー / マジックストリング
- ネストの深さ、複雑度
- テスト容易性 (副作用の閉じ込め、依存注入)
- コメントの過不足 (「なぜ」を説明しているか)

**重要度の目安:**
- **高**: 何をしているか追えない、変更時に副作用が読めない
- **中**: 命名がミスリーディング、責務が曖昧
- **低**: 軽微なリファクタ余地、コメントの体裁

## 4. パフォーマンス (Performance)

ユーザー体験を損なう非効率がないか。

**チェック対象:**
- 不要な再レンダリング (React: 依存配列、`useMemo` / `useCallback` の妥当性)
- N+1 的な非効率なループ
- 巨大データの全件処理、メモリリーク
- バンドルサイズへの影響 (重い依存の追加)
- イベントハンドラ (resize, scroll, input) のデバウンス / スロットル要否

**重要度の目安:**
- **高**: 体感で分かるレベルの遅延、メモリ枯渇
- **中**: スケール時に問題化する非効率
- **低**: マイクロ最適化の余地

## 5. 一貫性 (Consistency)

既存コードベースの規約・パターンと整合しているか。

**チェック対象:**
- ESLint / Prettier ルールへの準拠
- 既存の命名規則・ファイル配置パターン
- 既存ヘルパー / コンポーネントの再利用 (車輪の再発明をしていないか)
- import 順序、相対パス vs エイリアス
- TypeScript の型定義スタイル (`interface` vs `type`)

**重要度の目安:**
- **高**: プロジェクトの基本方針に反する設計
- **中**: 既存パターンと食い違う実装
- **低**: スタイルの揺れ

---

## 重要度の総合判定ルール

複数観点で重要度が分かれた場合は、**最も高い重要度を採用**する。例：

- セキュリティ「中」+ 保守性「高」 → 総合「高」
- パフォーマンス「低」+ 一貫性「低」 → 総合「低」

ただし、**セキュリティ観点は常に 1 段階引き上げて評価**してよい。理由: 修正コストよりも事故リスクの方が大きいため。
